작성일: 2026-06-24

## 1. 전문가 그룹 논의 요약

이번 검토는 다음 관점으로 나누어 진행했다.

| 관점 | 핵심 질문 | 결론 |
|---|---|---|
| 진로·취업 교육/HRD | 학생이 실제로 막히는 지점은 무엇인가 | 직무명은 알지만 실제 업무, 필요 역량, 학습 순서, 포트폴리오 산출물을 모른다 |
| 기계·전자 산업 도메인 | 기계공학/전자공학 학생이 많이 가는 산업과 직무는 무엇인가 | 기업명보다 산업군-직무군-역량 중심으로 시작해야 정확도와 유지보수가 높다 |
| 제품·기술 구현 | 어떤 MVP가 빠르게 검증 가능한가 | 링크 모음이 아니라 진단, 갭 분석, 학습 로드맵, 자료 큐레이션이 연결된 도구가 필요하다 |
| 콘텐츠 품질·운영 리스크 | 공개 자료를 어떻게 안전하게 다룰 것인가 | 저작권, 최신성, 자료 품질, 부정확한 직무 매핑을 관리하는 검수 프로세스가 핵심이다 |

공통 결론은 명확하다. 이 서비스는 단순한 유튜브/강의 링크 모음이 아니라 `직무 선택 -> 역량 진단 -> 공개자료 큐레이션 -> 4주 학습 로드맵 -> 포트폴리오 산출물`로 이어지는 직무역량 로드맵 서비스로 포지셔닝해야 한다.

## 2. 서비스 방향성

### 2.1 한 문장 정의

기계공학과 전자공학 대학생이 관심 산업과 직무를 선택하면, 해당 직무에 필요한 역량과 검수된 공개 학습자료, 4주 학습 로드맵, 포트폴리오 산출물 예시를 제공하는 취업 준비 지원 서비스.

### 2.2 해결하려는 문제

| 문제 | 설명 |
|---|---|
| 직무 이해 부족 | 생산기술, 설비기술, 품질, 회로설계, 펌웨어, 반도체 공정 같은 직무명이 실제로 어떤 업무인지 모호함 |
| 학습 우선순위 혼란 | 전공수업, 자격증, 코딩, 프로젝트, 유튜브 강의 중 무엇을 먼저 해야 할지 판단하기 어려움 |
| 자료 탐색 피로 | 공개 자료는 많지만 난이도, 신뢰도, 직무 관련성을 학생이 직접 판별해야 함 |
| 전공과 취업의 단절 | 전공과목과 기업 직무역량이 어떻게 연결되는지 보이지 않음 |
| 포트폴리오 부족 | 강의를 듣고 끝나며, 자기소개서나 면접에서 증명할 산출물로 연결되지 않음 |

### 2.3 핵심 가치 제안

1. 학생에게 직무별 실제 업무, 필요 역량, 사용하는 도구, 대표 산출물을 보여준다.
2. 현재 수준을 행동 기준으로 진단해 부족한 역량을 알려준다.
3. 검수된 공개 자료를 직무역량, 난이도, 예상 시간, 산출물 기준으로 추천한다.
4. 4주 또는 8주 단위의 실행 가능한 학습 로드맵을 제공한다.
5. 학습 결과를 포트폴리오, GitHub, 분석보고서, 회로도, CAD 모델, 해석 리포트 등으로 남기게 한다.

## 4. 직무역량 분류체계

초기에는 기업별 모든 직무를 직접 매핑하지 않는다. 같은 회사라도 사업부와 팀에 따라 요구역량이 다르기 때문이다. MVP에서는 산업군과 대표 직무군을 기준으로 공통 역량을 설계하고, 이후 채용공고와 현직자 검토로 기업별 차이를 보정한다.

### 4.1 MVP 직무 트랙

| 트랙 | 주요 대상 | 대표 직무 |
|---|---|---|
| 기계 설계·CAE | 기계공학 | 기구설계, 제품개발, 구조해석, 열유체 해석 |
| 생산·공정·품질 | 기계/전자 공통 | 생산기술, 공정기술, 품질관리, 신뢰성, 시험평가 |
| 반도체 공정·장비 | 기계/전자 공통 | 공정기술, 설비기술, 장비기술, 패키징, 테스트 |
| 전자회로·PCB·하드웨어 | 전자공학 | 회로설계, PCB, 전장 하드웨어, 검증 |
| 임베디드·제어 | 전자/기계 공통 | 펌웨어, MCU, RTOS, 제어, 로봇, 자동차 전장 |

2차 확장 후보는 배터리 팩·BMS·열관리, 전력전자·모터제어, RF·통신, 로봇·스마트팩토리, 플랜트·설비 엔지니어링이다.

## 6. MVP 사용자 경험

첫 화면은 마케팅 랜딩페이지가 아니라 바로 진로 탐색 도구여야 한다.

### 6.1 핵심 사용자 흐름

1. 학생이 전공, 학년, 관심 산업, 관심 직무, 이수 과목, 사용 가능한 도구, 프로젝트 경험을 입력한다.
2. 서비스가 관련 직무 트랙 3개를 추천하거나 사용자가 직접 선택한다.
3. 선택한 직무 트랙의 실제 업무, 필요 전공지식, 사용하는 도구, 대표 산출물, 흔한 오해를 보여준다.
4. 학생이 행동 기준 체크리스트로 현재 역량을 진단한다.
5. 서비스가 부족한 역량과 우선순위를 보여준다.
6. 4주 학습 로드맵을 제안한다.
7. 각 주차마다 공개자료, 실습, 산출물 체크리스트를 제공한다.
8. 학생은 관심 직무, 자료, 완료 항목, 산출물을 저장한다.

### 6.2 화면 구성

| 화면 | MVP 기능 |
|---|---|
| 직무 탐색 | 직무 카드, 산업 필터, 전공 필터, 준비 난이도, 사용하는 도구 |
| 직무 상세 | 실제 업무, 필요 역량, 관련 전공과목, 대표 프로젝트, 오해와 주의점 |
| 역량 진단 | 행동 기준 체크리스트, 보유 역량, 부족 역량 |
| 로드맵 | 4주 학습 계획, 주차별 자료, 실습, 산출물 |
| 자료 라이브러리 | 검색, 필터, 난이도, 자료 유형, 검수상태 |
| 내 계획 | 북마크, 진행률, 완료 체크, 산출물 메모 |
| 관리자 | 자료 등록, 검수상태 변경, 링크 점검, 피드백 확인 |

## 7. 타당성 검증 계획

### 7.1 검증 가설

| 가설 | 검증 방법 |
|---|---|
| H1. 직무별 실제 업무와 산출물을 보여주면 직무 이해도가 오른다 | 사용 전/후 직무 이해도 5점 척도 |
| H2. 역량 진단을 하면 학습 우선순위가 명확해진다 | 진단 완료 후 "무엇부터 할지 알겠다" 응답률 |
| H3. 검수된 공개자료 큐레이션은 무작위 검색보다 유용하다 | 추천자료 만족도, 자료 클릭률, 저장률 |
| H4. 4주 로드맵은 실제 학습 행동으로 이어진다 | 로드맵 시작률, 1주/4주 재방문율, 완료율 |
| H5. 산출물 중심 안내는 포트폴리오 작성에 도움이 된다 | 산출물 제출 수, 자기소개서/면접 활용 사례 |

### 7.2 파일럿

| 항목 | 계획 |
|---|---|
| 대상 | 기계공학/전자공학 2~4학년 30~50명 |
| 기간 | 4주 |
| 직무 트랙 | 기계 설계·CAE, 생산·공정·품질, 반도체 공정·장비, 전자회로·PCB, 임베디드·제어 |
| 운영 방식 | 사전 설문 -> 진단 -> 4주 로드맵 수행 -> 산출물 제출 -> 사후 설문/인터뷰 |
| 주요 지표 | 진단 완료율, 로드맵 저장률, 추천자료 클릭률, 1주 재방문율, 4주 산출물 제출률 |
| 정성 질문 | 이 사이트가 없었다면 같은 자료를 찾을 수 있었는가, 추천 로드맵이 행동으로 이어졌는가, 직무 이해도가 높아졌는가 |

### 8.2 추천 방식

MVP에서는 복잡한 AI 추천보다 규칙 기반 추천으로 시작한다.

1. 사용자의 전공, 학년, 관심 산업, 관심 직무를 받아 관련 RoleTrack을 후보로 만든다.
2. RoleTrack에 연결된 필수 Skill과 사용자의 진단 결과를 비교한다.
3. 부족한 Skill 중 priority가 높은 순서로 학습 우선순위를 만든다.
4. 각 Skill에 연결된 Resource 중 `verified` 또는 `reviewed` 상태의 자료를 우선 노출한다.
5. 난이도는 사용자 수준보다 한 단계 높은 자료까지 허용한다.
6. LearningPath가 있는 경우 주차별로 묶어서 보여준다.

## 9. 기술 스택 제안

### 9.1 MVP 권장 스택

| 영역 | 제안 |
|---|---|
| 프론트엔드 | 무료 파일럿은 의존성 없는 정적 PWA, 이후 Next.js/TypeScript 확장 |
| 백엔드 | Next.js API Routes 또는 Supabase |
| 데이터베이스 | PostgreSQL, 초기에는 Supabase 또는 SQLite/Prisma도 가능 |
| 인증 | MVP에서는 선택 사항. 진행관리 저장이 필요하면 Supabase Auth 또는 Clerk |
| 검색 | 초기에는 DB full-text와 필터. 자료가 늘어나면 Meilisearch, Typesense, Algolia 검토 |
| 배포 | Cloudflare Pages 우선. Next.js 서버 기능이 많아지면 OpenNext/Cloudflare Workers 또는 Vercel 대안 검토 |
| 콘텐츠 운영 | 초기에는 CSV/Google Sheets/Airtable seed import, 이후 관리자 페이지 |
| 분석 | Vercel Analytics, PostHog, Plausible 중 하나 |

### 9.2 구현 원칙

1. 콘텐츠와 UI를 분리한다. 자료/직무/역량 데이터는 DB 또는 JSON seed로 관리한다.
2. 자료 추천은 처음부터 AI 자동 생성에 의존하지 않는다.
3. 모든 자료는 검수상태와 최종확인일을 가진다.
4. 사용자의 개인정보는 최소 수집한다.
5. 기업별 세부 직무 매핑은 검증 전까지 `검토 필요` 상태로 표시한다.

### 9.3 Cloudflare 배포 방향

특별한 제약이 없다면 MVP는 Cloudflare Pages를 기본 배포 대상으로 둔다. 초기 화면과 자료 탐색, 진단, 로드맵 기능은 정적 페이지와 클라이언트 상태만으로도 충분히 구현할 수 있으므로 Cloudflare Pages와 잘 맞는다.

| 항목 | 권장 방향 |
|---|---|
| 프론트엔드 배포 | Cloudflare Pages |
| 도메인 연결 | Cloudflare DNS에서 커스텀 도메인 연결 |
| API 기능 | 초기에는 Supabase 직접 연동 또는 Pages Functions 사용 |
| 인증/DB | Supabase Auth/PostgreSQL을 우선 검토 |
| 서버 렌더링 | 꼭 필요할 때만 OpenNext 기반 Cloudflare Workers 배포 검토 |
| 캐싱 | 공개 자료 목록, 직무 트랙, 역량 데이터는 CDN 캐싱 활용 |
| 주의점 | Node.js 전용 패키지, 파일시스템 의존 API, 긴 서버 작업은 Cloudflare 런타임 제약을 확인해야 함 |

구현 전략은 Cloudflare 친화적으로 단순하게 시작하는 것이다. MVP 단계에서는 데이터 seed를 JSON/CSV 또는 Supabase에 두고, 사용자 진행상황은 로컬 스토리지 또는 Supabase에 저장한다. 이렇게 하면 Cloudflare Pages에서 빠르게 공개하고, 나중에 관리자 기능이나 추천 API가 커질 때 Workers/Functions로 확장할 수 있다.

무료 파일럿에서는 휴대폰 앱 검증을 위해 네이티브 앱보다 PWA를 우선한다. PWA는 Cloudflare Pages에서 무료로 배포할 수 있고, 학생은 모바일 브라우저에서 홈 화면에 추가해 앱처럼 사용할 수 있다. 네이티브 앱은 파일럿 지표로 반복 사용과 저장 니즈가 확인된 뒤 검토한다.

## 11. 콘텐츠 품질·저작권·운영 리스크

### 11.1 저작권과 링크 정책

1. 원문을 복사하지 않고 제목, 요약, 추천 이유, 원본 링크를 제공한다.
2. 유튜브는 공식 임베드가 허용되는 경우에만 임베드한다.
3. 기업 로고, 상표, 강의 이미지, 썸네일 사용은 출처별 정책을 확인한다.
4. 유료 강의나 폐쇄형 자료를 무단 요약/재배포하지 않는다.
5. 자료별 `licenseNote`와 `lastVerifiedAt`을 남긴다.
6. 사용자가 업로드하는 포트폴리오가 생기기 전까지 개인정보와 저작물 수집을 최소화한다.

### 11.2 자료 품질 기준

| 기준 | 설명 |
|---|---|
| 출처 신뢰도 | 대학, 공식 문서, 정부/협회, 기업 기술자료, 검증된 교육 채널 우선 |
| 직무 관련성 | 특정 직무역량과 직접 연결되는가 |
| 난이도 적합성 | 입문자에게 너무 어렵거나 지나치게 얕지 않은가 |
| 실행 가능성 | 학습 후 실습 또는 산출물로 이어지는가 |
| 최신성 | 기술, 표준, 도구 버전이 지나치게 오래되지 않았는가 |
| 편향성 | 특정 유료교육이나 기업 홍보로 과도하게 치우치지 않는가 |

### 11.3 피해야 할 부정확한 매핑

| 피해야 할 매핑 | 이유 |
|---|---|
| 기계공학 = CAD만 배우면 됨 | 설계에는 재료, 공차, 제조공정, 시험, 원가 이해가 함께 필요함 |
| CAE = 소프트웨어 버튼 사용법 | 해석의 핵심은 물리 모델링, 경계조건, 결과 검증 |
| 생산기술 = 단순 현장관리 | 공정 최적화, 자동화, 품질, 설비 데이터 분석이 중요 |
| 품질 = 꼼꼼함 | 통계, 계측, FMEA, 원인분석 방법론이 필요 |
| 회로설계 = 납땜/아두이노 | 제품 회로설계는 전원, 노이즈, 신뢰성, 부품 선정, 검증이 핵심 |
| 임베디드 = 앱 개발 | 레지스터, 인터럽트, 실시간성, 통신 프로토콜, 하드웨어 디버깅이 중요 |
| 반도체 공정 = 전자회로 지식 | 공정/소자/장비는 물리, 화학, 재료, 진공, 플라즈마 이해가 필요 |
| AI 역량 = 모든 직무의 최우선 역량 | 데이터 활용은 중요하지만 도메인 물리와 실험 이해를 대체하지 못함 |
| 회사명 기준으로 역량 고정 | 같은 회사라도 사업부와 팀에 따라 요구 역량이 다름 |

## 14. 결론

프로젝트는 타당하다. 다만 성공 여부는 자료를 많이 모으는 데 있지 않고, 직무와 역량, 학습자료, 산출물을 정확히 연결하는 데 있다. 초기에는 기업별 세부 직무를 과하게 약속하지 말고, 검증 가능한 대표 직무 트랙 5개와 공개자료 50~100개로 시작해야 한다.

가장 중요한 제품 원칙은 다음이다.

1. 직무명보다 실제 업무와 산출물 중심으로 설명한다.
2. 자료 추천은 항상 역량과 연결한다.
3. 학습은 반드시 포트폴리오 산출물로 끝나게 한다.
4. 자료와 직무 매핑에는 검수상태와 최신성 정보를 표시한다.
5. AI 자동화는 콘텐츠 품질 검증 이후에 도입한다.
