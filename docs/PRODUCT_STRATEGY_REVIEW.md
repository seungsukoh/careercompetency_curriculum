# 제품 진행 전략 검토

> 참고: 이 문서는 초기 5개 트랙 MVP 검토 내용을 포함한다. 2026-06-29 현재 구현 기준은 16개 직무군, 96개 세부 직무, 123개 교육자료이며, 최신 우선순위와 검증 기준은 `docs/PM_ACTION_PLAN.md`와 `docs/REQUIREMENTS_VALIDATION_CRITERIA.md`를 따른다.

## 결론

1차 제품은 교수자/관리자 도구가 아니라 학생 개인이 혼자 공부하기 위해 검수된 자료를 받고, 그 자료가 학습 순서대로 배열된 짧은 학습 계획을 보는 도구로 시작한다.

초기 포지셔닝:

> 기계공학/전자공학 학생이 관심 직무를 고르면, 한국어로 접근 가능한 공개 학습자료를 난이도와 선후관계에 맞춰 배열해 추천하고, 예상 소요시간과 산출물을 함께 보여주는 자기주도 학습 자료 큐레이션 도구.

## 전문가 관점 검토

### 진로교육/HRD 관점

학생은 보통 "무엇을 공부해야 하는지"보다 먼저 "내가 어떤 직무를 준비하는지"를 모른다. 따라서 자료 추천 전에 직무 트랙을 너무 복잡하게 만들지 말고, 대표 직무 5개 안에서 시작해야 한다.

필요한 기능:

- 직무별 실제 업무 설명
- 필요한 역량을 학생 언어로 풀어쓴 체크리스트
- 입문자가 바로 볼 수 있는 한국어 자료
- 영상 시청 후 남길 산출물 안내

### 공학 교수자 관점

학생은 본인이 부족한 전공 개념을 정확히 진단하기 어렵다. 교수자 관점에서는 자료 추천이 전공 기초와 직무 역량을 연결해야 한다.

필요한 기능:

- 직무별 선행 전공 개념 표시
- 학생이 체크하지 못한 역량에 맞춘 보완 자료 추천
- 단순 영상 목록이 아니라 학습 순서 제안
- 과제나 상담에 활용할 수 있는 결과 복사/공유 기능

### 산업/채용 관점

채용에서는 "강의를 들었다"보다 "무엇을 해봤고 어떤 산출물을 남겼는지"가 중요하다. 따라서 모든 자료 추천은 포트폴리오 산출물로 이어져야 한다.

필요한 기능:

- 자료별 expectedOutput 필드
- 직무별 대표 산출물 예시
- 1주/2주/4주 단위의 작은 완성 목표
- 면접에서 설명 가능한 결과물 중심 로드맵

### 제품 관점

초기 제품은 추천 정확도보다 신뢰와 완주율이 중요하다. 자료를 많이 넣기보다, 적은 자료라도 왜 이 자료를 지금 봐야 하는지 설명해야 한다.

필요한 기능:

- 한국어 자료 우선
- 조회수/댓글 등 참여 신호 기반 후보 선별
- 난이도, 선행조건, 예상 소요시간 표시
- 짧은 학습 계획 생성

### SW 아키텍처 관점

무료 파일럿은 정적 PWA로 시작하되, 나중에 계정, DB, 유료 기능, 교수자 기능으로 옮기기 쉬운 데이터 경계를 유지해야 한다. 초기부터 복잡한 백엔드를 만들 필요는 없지만, 자료/직무/역량/진행상태의 식별자와 필드 구조는 마이그레이션을 전제로 잡아야 한다.

필요한 설계:

- 자료, 직무, 역량, 로드맵 데이터의 ID 체계 유지
- `app.js`에 들어 있는 seed 데이터를 나중에 `data/*.json` 또는 DB 테이블로 분리 가능하게 구성
- `localStorage` 진행상태를 나중에 `user_progress` 테이블로 옮길 수 있게 상태 구조 유지
- Cloudflare Pages 정적 배포에서 Pages Functions, Workers, Supabase 등으로 확장 가능한 경계 유지
- PWA 캐시 버전과 데이터 버전을 분리해 자료 업데이트가 반영되도록 설계

### SW 개발자 관점

초기 개발은 기능을 늘리기보다 검증 가능한 작은 흐름을 안정적으로 완성하는 것이 우선이다. 코드 구조는 단순해야 하고, 자료 모델이 바뀌어도 UI 수정 범위가 작아야 한다.

필요한 구현:

- 자료 카드가 `estimatedMinutes`, `practiceMinutes`, `sequenceLevel`, `prerequisites`, `expectedOutput`, `qualityStatus`를 표시
- 추천 자료를 학습 순서대로 정렬
- 학생이 수동으로 저장/완료 체크할 수 있는 상태 저장
- 모바일 화면에서 버튼과 텍스트가 깨지지 않도록 검증
- 브라우저 검증 후 작게 커밋 가능한 상태 유지

## 학생 입장에서 필요한 것

학생이 직접 말할 수 있는 니즈:

- 어떤 직무가 나에게 맞는지 알고 싶다.
- 뭘 먼저 공부해야 할지 모르겠다.
- 유튜브나 강의가 너무 많아서 고르기 어렵다.
- 시간이 부족하니 단기간에 할 수 있는 계획이 필요하다.
- 포트폴리오나 면접에 쓸 수 있는 결과물을 만들고 싶다.

학생이 아직 모를 수 있는 니즈:

- 직무별로 필요한 전공 기초가 다르다.
- 자료를 많이 보는 것보다 순서와 산출물이 중요하다.
- 인기 영상이 항상 좋은 학습 경로는 아니다.
- 영상만 봐서는 실무 역량으로 증명하기 어렵다.
- 본인의 부족한 역량을 먼저 좁혀야 시간 낭비가 줄어든다.

## 교수자 입장에서 도와줄 수 있는 것

초기에는 교수자 계정을 만들지 않는다. 대신 학생 결과를 교수자가 상담에 활용할 수 있게 한다.

교수자 지원 방식:

- 학생이 선택한 직무 트랙 확인
- 진단 결과와 부족 역량 확인
- 추천 자료와 예상 학습시간 확인
- 산출물 목표 확인
- 상담 전에 학생의 준비 상태를 빠르게 파악

추후 확장:

- 교수자용 추천 템플릿
- 학과 과목과 직무역량 매핑
- 학년별 추천 로드맵
- 상담용 PDF/복사 결과 생성

## 단계별 진행

### 1단계: 자기주도 자료 탐색 MVP

목표: 학생이 혼자 들어와서 직무를 고르면, 서비스가 검수해 둔 자료를 학습 순서대로 제공한다.

포함:

- 직무 트랙 5개
- 한국어 자료 우선
- 자료별 난이도, 예상시간, 선행조건, 산출물
- 조회수/댓글 등 참여 신호 참고
- 1주/2주/4주 추천 경로

제외:

- 교수자 계정
- 관리자 페이지
- AI 자동 생성
- 전체 기업/직무 매핑

### 2단계: 학생 상황별 커리큘럼

목표: 학생의 시간, 목표, 수준에 따라 자료 범위를 좁히거나 넓힌다.

포함:

- 주당 가능 시간
- 준비 기간
- 목표 선택: 직무 이해, 전공 보완, 면접 준비, 포트폴리오
- 짧은 계획과 긴 계획 전환

### 3단계: 교수자 추천 보조

목표: 교수자가 학생 상담이나 과목 안내에 활용한다.

포함:

- 상담용 요약
- 학년별 추천 경로
- 학과 과목과 직무역량 연결
- 학생 결과 공유

## 바로 다음 제품 작업

1. 자료 데이터 모델 확장:
   - `language`
   - `estimatedMinutes`
   - `practiceMinutes`
   - `sequenceLevel`
   - `prerequisites`
   - `engagement`
   - `expectedOutput`

2. 화면에 추가 표시:
   - 예상 소요시간
   - 추천 학습 순서
   - 한국어 자료 여부
   - 산출물

3. 한국어 자료 후보 수집:
   - 트랙별 최소 5개
   - 총 25개부터 시작
   - 후보 상태는 `candidate`, 검수 후 `reviewed`

4. 파일럿 검증:
   - 학생 5명에게 자료 추천이 실제로 도움이 되는지 확인
   - "이 자료를 왜 먼저 봐야 하는지 이해했는가"를 핵심 질문으로 둔다.

## 운영 모델

운영자가 지속적으로 많은 자료를 관리하는 방식은 피한다. 1차 제품은 트랙별로 작고 안정적인 한국어 자료 묶음을 만들고, 학생에게 그 자료를 학습 순서대로 제공하는 방식으로 시작한다.

운영 부담을 낮추는 원칙:

- 자료 수보다 자료의 안정성과 설명 품질을 우선한다.
- 트랙별 5~10개 핵심 자료로 시작한다.
- 최신성이 낮은 기초 자료와 공식 공개강의를 우선 사용한다.
- 자료 검수 주기는 주간이 아니라 월간 또는 학기 단위로 잡는다.
- 학생 피드백과 링크 오류 신고를 통해 보완 대상을 찾는다.
- 조회수, 댓글 수, 링크 생존 여부는 추후 자동 점검 대상으로 둔다.
- 추천 순서와 산출물 판단은 당분간 사람 검수 기준을 유지한다.
