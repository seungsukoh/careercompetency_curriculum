# 아키텍처 결정 기록

## ADR-001: 1차 제품은 정적 PWA로 유지한다

결정: 초기 파일럿은 의존성 없는 정적 PWA로 유지한다.

이유:

- 배포와 운영 비용이 낮다.
- 학생 파일럿에 충분하다.
- 인증, DB, 결제, 관리자 기능을 premature하게 만들지 않아도 된다.

결과:

- Cloudflare Pages에 적합하다.
- 사용자 진행상태는 우선 `localStorage`에 저장한다.
- 서버 기능은 파일럿 검증 후 도입한다.

## ADR-002: 자료 데이터는 UI와 분리 가능한 구조로 관리한다

결정: 현재는 `app.js` 안에 seed 데이터가 있지만, 필드는 나중에 `data/*.json` 또는 DB로 분리 가능한 형태로 유지한다.

필수 필드:

- `id`
- `title`
- `url`
- `sourceType`
- `provider`
- `language`
- `difficulty`
- `sequenceLevel`
- `estimatedMinutes`
- `practiceMinutes`
- `prerequisites`
- `tracks`
- `skills`
- `reason`
- `expectedOutput`
- `qualityStatus`
- `engagement`

이유:

- 자료 큐레이션과 앱 개발을 병렬로 진행할 수 있다.
- 나중에 Supabase/PostgreSQL로 마이그레이션하기 쉽다.
- 자료 업데이트가 앱 로직 변경으로 번지지 않는다.

## ADR-003: 1차 추천은 AI 자동 생성이 아니라 검수된 큐레이션이다

결정: 초기 추천은 AI 자동 생성이 아니라 사람이 기준을 정한 자료 묶음과 순서 기반으로 제공한다.

이유:

- 학생에게 잘못된 학습 순서를 제시할 위험을 줄인다.
- 자료 품질과 저작권 리스크를 통제한다.
- 작은 파일럿에서 추천 정확도보다 신뢰가 중요하다.

결과:

- 자료 수를 많이 늘리지 않는다.
- 트랙별 5~10개 핵심 자료로 시작한다.
- AI/자동화는 링크 확인, 조회수/댓글 갱신 같은 보조 기능부터 검토한다.

## ADR-004: 운영자는 편집자 역할로 제한한다

결정: 운영자가 매주 자료를 찾아다니는 구조는 피한다. 운영자는 후보를 유지/숨김/교체 필요로 판단하는 편집자 역할을 맡는다.

이유:

- 운영 부담이 높으면 서비스가 유지되지 않는다.
- 자료의 최신성보다 안정적인 기본 학습 경로가 1차 파일럿에 더 중요하다.

결과:

- 검수 주기는 월간 또는 학기 단위로 시작한다.
- 학생 피드백과 링크 오류 신고를 통해 보완 대상을 찾는다.
- 나중에 관리 화면을 만들더라도 클릭 기반 승인/거절 중심으로 설계한다.

## ADR-005: 유료화는 데이터 경계만 대비하고 기능은 보류한다

결정: 유료화, 계정, 교수자 대시보드, 결제 기능은 지금 만들지 않는다. 다만 데이터 모델은 추후 이전 가능하게 유지한다.

유료화 후보:

- 개인 맞춤 장기 커리큘럼
- 학습 기록 동기화
- 포트폴리오 산출물 가이드
- 교수자/기관용 상담 리포트

보류 이유:

- 아직 학생이 자료 순서 추천을 실제로 원하는지 검증되지 않았다.
- 인증/결제/DB는 유지보수 비용을 크게 올린다.

## ADR-006: 병렬 개발은 파일 소유권으로 관리한다

결정: 병렬 작업은 역할명이 아니라 파일 소유권으로 나눈다.

기준:

- PM/전략: `docs/PRODUCT_STRATEGY_REVIEW.md`, `docs/PM_ACTION_PLAN.md`
- 자료 큐레이션: `docs/RESOURCE_CURATION_POLICY.md`, 추후 `data/resources.json`
- 앱 UI/UX: `index.html`, `styles.css`, `app.js`
- QA/운영: `README.md`, `docs/QA_CHECKLIST.md`, `sw.js`

이유:

- 같은 파일을 동시에 수정하면 충돌과 회귀가 늘어난다.
- 작은 팀/단독 작업에서도 다음 세션 인수인계가 쉬워진다.

## 유지보수 원칙

1. 새 기능보다 데이터 구조의 일관성을 우선한다.
2. 사용자가 확인한 문제만 기능으로 승격한다.
3. 자료 수를 늘리기보다 추천 이유와 순서를 선명하게 만든다.
4. 정적 파일럿에서 해결 가능한 문제는 백엔드를 만들지 않는다.
5. 운영자가 반복해서 해야 하는 일은 만들지 않거나 나중에 자동화한다.
