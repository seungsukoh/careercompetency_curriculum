# 요구사항 기반 검증 기준

## 목적

이 문서는 개발 속도보다 방향성과 유지보수성을 우선하기 위한 기준 문서다.

앞으로 기능, 자료, UI, 운영 절차를 추가할 때는 먼저 요구사항에 연결하고, 그 요구사항을 검증할 수 있는 기준을 통과해야 한다. 검증 기준은 취향이나 추측이 아니라 정리된 요구사항에서 나온다.

## 제품 범위 기준

1차 제품은 학생이 혼자 공부할 때 사용할 `직무 기반 학습자료 큐레이션 도구`다.

포함한다:

- 학생이 직무 트랙을 고른다.
- 앱이 직무별 핵심 역량과 학습 순서를 보여준다.
- 검수된 한국어 접근 가능 자료를 우선 제공한다.
- 자료마다 난이도, 선행조건, 예상 시간, 산출물을 보여준다.
- 학생이 자료 저장과 완료 체크를 수동으로 할 수 있다.
- 교수자나 상담자는 학생의 선택, 진단, 저장/완료 상태를 요약으로 참고할 수 있다.

지금은 포함하지 않는다:

- 로그인, 결제, 유료 기능
- 교수자 관리자 화면
- AI 자동 커리큘럼 생성
- 전체 웹 자동 크롤링
- 기업별 모든 직무 세부 매핑
- 학생 개인정보와 산출물 파일 수집

## 요구사항 목록

| ID | 요구사항 | 우선순위 | 검증 질문 |
|---|---|---|---|
| R1 | 학생은 자신의 전공/학년/관심 산업에 맞는 직무 트랙을 선택할 수 있어야 한다. | P0 | 학생이 첫 화면에서 어떤 직무를 고를지 1분 안에 판단할 수 있는가 |
| R2 | 직무 트랙은 실제 업무, 필요 역량, 도구, 대표 산출물을 설명해야 한다. | P0 | 학생이 직무명을 넘어 실제로 무엇을 하는지 설명할 수 있는가 |
| R3 | 역량 진단은 행동 기준 체크리스트여야 한다. | P0 | 학생이 "안다/모른다"가 아니라 "해본 적 있다/설명할 수 있다/산출물이 있다"로 답할 수 있는가 |
| R4 | 자료 추천은 단순 링크 모음이 아니라 학습 순서가 있는 큐레이션이어야 한다. | P0 | 첫 번째로 볼 자료와 그다음 자료의 이유가 납득되는가 |
| R5 | 자료 카드는 난이도, 선행조건, 예상 시간, 실습 시간, 산출물, 검수상태를 가져야 한다. | P0 | 학생이 오늘 할 수 있는 분량인지 판단할 수 있는가 |
| R6 | 한국어로 접근 가능한 자료를 우선 사용해야 한다. | P0 | 한국어 자료만으로도 1차 파일럿 흐름을 시작할 수 있는가 |
| R7 | YouTube 자료는 조회수/댓글 같은 참여 신호를 후보 선별에 쓰되, 최종 판단은 직무 적합성/공식성/최신성/산출물 연결성으로 해야 한다. | P1 | 인기 영상이지만 학습경로에 맞지 않는 자료를 제외할 수 있는가 |
| R8 | 학생은 자료 저장과 완료 체크를 수동으로 할 수 있어야 한다. | P0 | 진행 상태가 새로고침 후에도 유지되는가 |
| R9 | 로드맵 주차별 완료 체크가 가능해야 한다. | P0 | 학생이 이번 주 학습이 끝났는지 스스로 표시할 수 있는가 |
| R10 | 파일럿 결과 요약은 상담이나 피드백에 쓸 수 있어야 한다. | P0 | 선택 트랙, 진단 결과, 저장/완료 자료, 완료 주차가 복사되는가 |
| R11 | 운영자가 매주 많은 자료를 직접 찾아야 하는 구조를 피해야 한다. | P0 | 월 1회 또는 학기별 점검으로 유지 가능한가 |
| R12 | 데이터 구조는 나중에 JSON/DB/유료 기능으로 이전하기 쉬워야 한다. | P1 | 자료/직무/진행상태가 UI 코드와 과하게 섞이지 않는가 |
| R13 | MVP는 정적 PWA로 유지해야 한다. | P0 | 백엔드 없이 배포와 사용이 가능한가 |
| R14 | 저작권과 출처 리스크를 줄여야 한다. | P0 | 원문 복제 없이 제목, 요약, 추천 이유, 원본 링크 중심으로 제공하는가 |
| R15 | 파일럿 성공 여부를 소수 학생 테스트로 판단할 수 있어야 한다. | P0 | 5명 테스트에서도 선택률, 진단 완료, 저장/완료, 정성 피드백을 확인할 수 있는가 |

## 인수 기준

### P0 통과 기준

P0 요구사항은 파일럿 전에 모두 통과해야 한다.

| 영역 | 통과 기준 |
|---|---|
| 직무 선택 | 5개 트랙이 보이고, 사용자가 하나를 선택하면 상세/진단/로드맵/자료 화면이 같은 트랙 기준으로 바뀐다. |
| 직무 이해 | 각 트랙에 실제 업무, 필요 역량, 도구, 산출물, 흔한 오해가 표시된다. |
| 진단 | 체크리스트를 선택하면 점수와 부족 항목이 즉시 갱신된다. |
| 자료 추천 | 자료는 학습 순서, 난이도, 한국어 접근성, 예상 시간, 선행조건, 산출물 기준으로 표시된다. |
| 진행 체크 | 자료 저장/완료와 로드맵 완료가 새로고침 후에도 유지된다. |
| 요약 복사 | 파일럿 결과 복사 내용에 선택 트랙, 진단, 저장/완료 자료, 완료 주차가 포함된다. |
| 모바일 | 휴대폰 폭에서 주요 버튼과 카드 텍스트가 겹치지 않고 핵심 흐름을 완료할 수 있다. |
| 운영 | 자료 검수상태와 최종 확인일 또는 확인 예정일을 기록할 수 있는 필드가 있다. |
| 배포 | 정적 파일만으로 로컬 서버와 Cloudflare Pages에서 동작할 수 있다. |

### P1 통과 기준

P1은 파일럿 중 또는 파일럿 직후 개선 기준이다.

| 영역 | 통과 기준 |
|---|---|
| 자료 품질 | 트랙별 한국어 자료 후보가 최소 5개 이상이고, `candidate/reviewed/verified` 상태가 구분된다. |
| 시간 추정 | 자료별 학습 시간과 실습 시간이 과도하게 낙관적이지 않다. |
| 순서 설명 | 왜 이 자료가 이 순서인지 추천 이유에 드러난다. |
| 학생 피드백 | "첫 자료가 명확했는가", "순서가 납득됐는가", "산출물이 도움이 됐는가" 질문을 수집한다. |
| 유지보수 | 자료 추가/수정이 UI 구조 변경 없이 가능하다. |

## 검증 방법

| 검증 유형 | 대상 | 방법 | 산출물 |
|---|---|---|---|
| 요구사항 리뷰 | PM/교수자/아키텍처 | 요구사항이 제품 범위와 맞는지 확인 | 이 문서의 요구사항 수정 |
| 데이터 검수 | 자료 큐레이션 | 자료 필드, 검수상태, 순서, 산출물 확인 | 자료 후보표 또는 seed 데이터 |
| 기능 QA | 앱 UI/UX | 로컬 브라우저에서 핵심 플로우 수행 | `docs/QA_CHECKLIST.md` 결과 |
| 모바일 QA | 앱 UI/UX | 휴대폰 폭에서 선택, 진단, 저장, 완료 확인 | 스크린샷 또는 체크 결과 |
| 운영성 리뷰 | PM/아키텍처 | 운영자가 반복해야 하는 일이 많은지 확인 | 운영 부담 항목과 보류 결정 |
| 파일럿 검증 | 학생/교수자 | 5명 이상 사용 후 지표와 정성 질문 확인 | 파일럿 결과 요약 |

## 변경 승인 규칙

새 기능이나 자료를 추가하기 전 확인한다.

1. 어떤 요구사항 ID를 만족하는가?
2. P0/P1/P2 중 어디에 속하는가?
3. 검증 방법이 있는가?
4. 운영자가 반복해서 관리해야 하는 일이 늘어나는가?
5. 나중에 DB/계정/유료 기능으로 이전할 때 데이터 경계를 깨지 않는가?

요구사항 ID에 연결되지 않는 작업은 기본적으로 보류한다. 단, 버그 수정과 보안/저작권 리스크 완화는 예외로 즉시 처리할 수 있다.

## 파일럿 성공 기준

초기에는 큰 지표보다 학생 5명의 실제 반응을 본다.

| 지표 | 목표 |
|---|---|
| 직무 트랙 선택 | 5명 중 4명 이상 |
| 역량 진단 완료 | 5명 중 3명 이상 |
| 추천 자료 저장 | 5명 중 3명 이상 |
| 자료 또는 로드맵 완료 체크 | 5명 중 2명 이상 |
| 첫 번째 학습자료가 명확했다는 응답 | 5명 중 3명 이상 |
| 추천 순서가 납득됐다는 응답 | 5명 중 3명 이상 |
| 산출물이 학습 동기에 도움이 됐다는 응답 | 5명 중 3명 이상 |

## 전문가 그룹 검토 관점

| 관점 | 검토 책임 |
|---|---|
| PM | 범위, 우선순위, 성공 기준, 보류할 기능 결정 |
| 진로/교수자 | 학생이 모르는 문제까지 드러나는지, 상담에 쓸 수 있는지 검토 |
| 산업 도메인 | 직무 설명과 역량 연결이 현실적인지 검토 |
| 자료 큐레이션 | 자료 품질, 순서, 시간, 산출물 연결성 검토 |
| SW 아키텍처 | 데이터 경계, 마이그레이션 가능성, 운영 부담 검토 |
| SW 개발 | 정적 PWA 구현 가능성, localStorage, 브라우저 호환성 검토 |
| QA | 요구사항별 통과/실패를 재현 가능하게 기록 |

## 다음 작업 판단

현재 다음 작업은 새 기능 개발이 아니라 다음 순서가 맞다.

2. 자료 후보를 수집할 때 R4~R7 기준으로만 선별한다.
3. 앱 검증은 R1~R15 요구사항별로 통과/실패를 표시한다.
4. 통과하지 못한 P0만 먼저 수정한다.
5. P1/P2 기능은 파일럿 결과가 나온 뒤 결정한다.
